1、前言

对运维而言，机器生命周期，指机器从到货上架通电那一刻起，到硬件故障且过保，厂商不予免费维修，需要下架报废那一刻为止。

这期间，运维工程师、运维经理、业务方(研发人员)参与了机器运维相关工作，

 

人员

工作或需求

运维工程师

机器分配、环境部署、服务部署、故障检测、故障维修

服务离线、服务恢复、扩容缩容、机器下线

运维经理

希望降低机器资源成本，提高资源利用率，机器操作效率尽可能地高

业务方

希望机器运维对业务透明，只看到机器资源 (CPU/MEM/IO/DISK)；

按需申请使用。

 

上述工作内容或业务需求，抽象出来可以分为5大类，

1.        故障管理

2.        变更管理

3.        环境管理

4.        容量管理

5.        服务管理

 

在传统运维里，机器归属天然是按业务来分的，既某个业务的运维工程师负责本业务机器的运维工作。这种划分方式存在以下问题，

n   在相同问题上重复投入人力，如机器故障维修，所有工程师都要做故障检测、服务离线、硬件维修、服务上线等相同的工作

n   由于业务的差异，工程师实现的方案不具备通用性，A业务机器维修流程无法在B业务使用，陷入了“既然无法复用，就重新做一遍同样的功能”的怪圈

n   工程师的主要工作是业务运维，机器运维只占一部分，没有精力去解决通用的容量管理、服务管理、长尾问题，自动化工具无法完全自动化，机器操作效率无法真正提高

这些问题，在大规模机器集群场景下尤为严重，一次常见的服务扩容，需要投入数个运维工程师数周的人力，各方对提高机器运维的需求非常迫切。

 

本文讲述了在大规模机器集群的环境下（约10万台量级），机器生命周期管理方案的设计和应用。

 

 

2、目标

从业务问题和需求出发，设计通用的机器生命周期管理系统，提高机器资源利用率和操作效率，解决业务方的痛点。

机器故障率，衡量了机器维修的效率、自动化程度。如果达标，说明无需投入人力，并且提高了机器资源利用率。

任务成功率，衡量了机器操作的效率、自动化程度。对于不同任务类型，有不同的成功率要求。例如，机器初始化、基础环境部署任务，用户希望交付时间<1天。

机器空闲率，衡量了机器资源的有效利用程度，机器每空跑一分钟都是在浪费金钱，所以要尽可能的避免空跑。

5、整体方案

5.1、机器运维模式的转变

 

 

 

 

针对传统运维模式存在的问题，在运维人员和机器之间加入了“机器管理系统”，消除机器环境、故障修复流程不同等差异，统一提供故障管理、环境管理、操作管理、容量管理、服务管理的功能组件，向运维人员屏蔽机器运维细节，提高机器操作效率、机器资源利用率；同时考虑了可扩展性，即新业务的机器管理，可低成本地纳入管理。

 

5.2、业界调研

调研了Google、阿里华佗、Microsoft AutoPilot 方案，详细对比见： 业界方案调研

对比总结

1、各家均采用 Master-Agent模式，即一套Master负责全局信息存储、决策控制， 管理的机器上都部署本机agent，负责采集信息、执行任务

2、各家具体实现不同，但核心组件类似，有：故障对象状态机、Master、Agent(Checker、Worker)、安全保护、故障数据分析预测

阿里华佗

Microsoft AutoPilot

AutoPilot 同样采用Master-Agent模式，设计目标包括服务部署和机器管理。

5.3、整体架构

 

 

 

l   基础组件负责采集机器故障、资源使用等信息，执行上层下发的任务；

l   工作流引擎负责生成既定规范的任务流描述，由状态机驱动，完成任务流程的控制和执行，形成操作闭环，提高自动化程度；

l   决策系统是通用运维逻辑的抽象，例如安全策略实现了并发控制、分级发布等功能，在机器维修、环境变更、批量机器操作都会使用到；

l   应用层是各类运维场景的入口，包括任务触发、数据展现等；

5.4.2、工作流引擎

工作流引擎包括任务及队列管理、任务模板、任务执行控制、状态机的功能。

 

对象定义：日常对机器的运维工作，通常是由一系列的顺序执行的操作任务组成，由此得到以下定义，

     1、Task，一个抽象的对机器的原子操作，如重装、修单盘、部署基础agent

     2、Job，一次完整的机器运维操作，从重装到拉起agent，检查服务状态

     3、一个Job包含一系列Task，且Task是按照顺序执行的；

     4、Group，用于组织机器批量操作任务，实现分级发布、并发控制

     5、Queue，Job生成后，放在Queue里，由Worker 执行；worker之间是互相独立；

     其中，Group、Queue、Worker是python 队列库 Celery 提供的功能；它们的关系如下图，

 

任务模板

 

任务模板定义，每个任务包含以下方法和属性，

1、start(), 开始执行

2、pause(), 暂停执行，取决于具体任务是否可暂停

3、resume(), 恢复执行

4、track_status(), 跟踪当前状态，new, handling, done, failed

5、done(), 标记本阶段任务为执行成功结束

6、fail(), 标记本阶段任务为执行失败结束

7、cancel(), 取消任务， 取决于取决于具体任务是否可取消

8、redo(), 重做任务

9、skip(), 跳过任务

10、第三方组件wrapper，封装了底层调用其他依赖系统细节

使用任务模板，可以将具体的实现和任务流程控制解耦，便于扩展。同时，一些通用的逻辑可以复用，例如状态检查、任务重试、抢锁等。

 

任务执行控制及队列

任务执行控制，支持以下特性，

1、每个Job与包含的Task关联通过有序数组来存储，可以控制每个task的执行顺序

2、根据task的状态选择不同的执行路径，包括执行下一个任务、等待、跳过、结束等

 

机器操作任务之间，需要互相独立，一个任务失败不能影响其他任务。

python的队列库Celery，功能满足此要求。Job生成放到Queue以后，每个Job由一个worker执行，worker是独立的进程，即使整个worker崩溃，也不会影响其他worker和Queue自身。

 

同时，Celery 提供Group来实现任务编排功能，满足分级发布、并发度控制要求。

1、一组任务，数量为N个，可配置为先执行1个，成功结束后执行N-1个；用于分级发布

2、一组任务，数量为N个，可配置每批2个，依次执行，直到完成；用于并发度控制

 

Celery支持多Queue，即可以按idc、app等维度将Job放入不同队列，做到Queue级别隔离，提高执行层的稳定性。

 

 

5.4.3、决策系统

l   Analyzer, 通过分析应用层提交的故障信息、服务信息，得出需要造作的对象，通常是机器、服务、环境实体

l   Checker，检查与操作对象相关的服务容量是否足够，变更中心是否有变更，线上整体是否有异常，如果不满足条件，则暂停任务，循环检查，直至符合条件

l   StrategyFilter，执行一系列的运维安全策略，包括对象去重、对象是否在提供服务、阈值拦截、并发控制、分级执行，保证操作是安全的

l   TaskGenerator，生产符合既定规范的任务描述文件，发送给工作流引擎执行

 

5.4.4、应用层

业务方对机器故障自动维修和日常操作的需求最为强烈，所以我们先开发了故障管理、变更管理子系统。

如下图所示，以故障维修场景为例，

 

l   从基础组件Master获取机器故障信息，包括机器死机、假死、CPU、内存、风扇、磁盘等硬件故障，发送给决策系统

l   决策系统根据故障信息，选择出操作对象，应用安全策略，最终生成任务描述文件，发送到工作流引擎执行

l   工作流引擎根据任务描述文件，调用对应的执行器，跟踪任务状态，直至任务进入最终态，形成操作闭环，减少人工介入次数
